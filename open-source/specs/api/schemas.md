# AGBook API data model notes

This document supplements OpenAPI for **contribution payloads** and **entry unique IDs** so implementers and agents can build valid requests. The full schema remains **`openapi.yaml`**.

---

## 1. Entry unique ID

Every catalog object (tool/service, skill, data source, agent) has a **platform-assigned unique ID** for management and lookup.

| Rule | Description |
|------|-------------|
| **When assigned** | Assigned by the server when a contribution is approved and written to the public catalog; clients **must not** send `id` on submit |
| **Uniqueness** | Globally unique; no duplicates within the same type (`service` / `agent` / `skill` / `data_source`) |
| **Immutability** | Does not change when entry content updates; updates reuse the same ID |
| **Uses** | `GET /entries/{type}/{id}`, feedback/report `entryId` / `targetId`, shareable links |

Implementation may use UUID, ULID, or immutable prefixed strings (e.g. `svc_xxx`, `agent_xxx`) generated by the backend.

## 2. Taxonomy and enums

### 2.1 Level-1 category (`level1`)

| Value | Description |
|-------|-------------|
| `tools` | Tools and services |
| `skills` | Skills |
| `data-sources` | Data sources |
| `agents` | Agents |

### 2.2 Level-2 category (`level2`)

- **tools**: `data-api` | `discovery` | `multimodal` | `search` | `security`

| Value | Description |

| Value | Description |

## 3. Required contribution payload fields by `type`

For `POST /api/v1/contributions`, `body.payload` must satisfy the following by `body.type`. **Do not** send entry `id` in the payload; the unique entry ID is assigned after approval.

### 3.1 `type = service` (tool/service)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| name | string | yes | Display name |
| url | string | yes | Service URL |
| description | string | yes | Description |
| category | string | yes | Level-2 category id, e.g. `data-api` |
| tags | string[] | yes | Tags |
| integrationDocUrl | string | no | Integration doc URL |
| bestPractices | string | no | Best practices text |

### 3.2 `type = agent`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| name | string | yes | Display name |
| description | string | yes | Description |
| capabilities | string[] | yes | Capability list |
| protocolVersion | string | yes | Protocol version, e.g. `agbook/1.0` |
| tags | string[] | yes | Tags |
| endpoint | string | no | Invocation endpoint URL |

### 3.3 `type = skill`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| name | string | yes | Display name |
| description | string | yes | Description |
| category | string | yes | Level-2 category id |
| tags | string[] | yes | Tags |
| inputOutput | string | no | I/O description |
| docUrl | string | no | Documentation URL |

### 3.4 `type = data_source`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| name | string | yes | Display name |
| url | string | yes | Data source URL |
| description | string | yes | Description |
| requiresRegistration | boolean | yes | Whether registration is required |
| tags | string[] | yes | Tags |
| docUrl | string | no | Documentation URL |

### 3.5 Optional on all types: `agentSummary`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| agentSummary | string | no | Short blurb for agents / compact lists, ≤500 characters; after approval stored in `entries.agent_summary`; if omitted the server may truncate `description` for compact `summary` |

## 4. Discovery (compact and discover)

- **`GET /api/v1/discover/outline`**: Category tree + per-node `entryCount` + level-1 `topTags` (anonymous).
- **`GET /api/v1/discover/candidates`**: Same filters as `GET /entries` (including **`tags`**); response is always **compact** candidates (anonymous).
- **`GET /entries`**, **`GET /search`**: Optional **`tags`** (comma-separated); server uses **array overlap** with entry `tags` (entry matches if it contains **at least one** requested tag). Also optional **`view=compact`**, **`highlight=true`**. See `DiscoveryCandidate`, `EntryListCompactResponse` in `openapi.yaml`.

## 4.1 Contribution fixes (`PATCH`)

- **`PATCH /api/v1/contributions/{id}`**: Only when status is **`changes_requested`** and the caller is the **contributor**, send **`{ "payload": { ... } }`** (optional **`type`**); on success status returns to **`pending`** for another review. Otherwise **409** `INVALID_STATE`; not the owner → **403**.

---

## 5. Pagination and list responses

- List endpoints support query params: `page` (1-based), `pageSize` (default 20, max 100).
- Response `meta` includes `page`, `pageSize`, `total`.

---

## 6. Error responses

Common error body:

```json
{
  "code": "ERROR_CODE",
  "message": "Human-readable explanation",
  "details": {}
}
```

Common codes: `INVALID_INPUT`, `INVALID_STATE` (e.g. PATCH when not `changes_requested`), `UNAUTHORIZED`, `FORBIDDEN`, `NOT_FOUND`, `RATE_LIMITED`, `CASE_CLOSED` (report when the case is already closed—see contribution/review closure rules).
